# モデル分担の型 v0.4.1

一行サマリ: **指令塔(従量課金の最上位モデル)は「短い入出力で高レバレッジの判断」だけに使い、コードの物量は枠内モデル(Opus/Sonnet)に流す。ダイヤルは設計の厚さで回し、検収は削らない。入力トークンは事前・事後とも可視化する。**

## コスト構造の前提
- サブスクプラン枠内のOpus/Sonnetは実質「使い放題の固定費」。従量になるのは**最上位モデル(以下、指令塔)だけ**。
- 最適化する変数はただ一つ「指令塔のトークンを何に使うか」。指令塔の価値が最大なのは出力が短くて判断の重い仕事=設計・依存関係の洗い出し・検収判定。
- 読み込みの分担は「安いモデルへ全振り」ではなく、下の**読み込み分担の分岐(ROI)**に従う(v0.4)。

## v0.4 改訂点(2026-07-05)
1. **入力トークンの可視化(事前・事後)**: 委譲先の「実測tok」単一値は定義が曖昧で判断を誤らせる。実測例(組込みロボット制御プロジェクトの1セッション): 委譲8体の**出力計138k**に対し**入力計10.6M**(非キャッシュ25k+キャッシュ書込1.1M+読取9.5M)— 出力だけ見ると2桁見誤る。以後、見積りは「入力/出力」を分けて提示し、実績はAgentツールのoutput_file(JSONL)を `skills/orch/scripts/agent_usage.py` で集計して記録する。
2. **委譲先の実測帯(n=8)**: 実装系1WP=入力0.5〜1.5M(キャッシュ込)/出力10〜30k、APIコール13〜28回。テスト反復が多い実装は入力4.4Mの実測例あり。調査系(Sonnet読み切り)=入力0.4〜1.4M/出力8〜19k。枠内モデルなら金額ゼロだが、レイテンシと(指令塔委譲・API直委譲では)コストに直結する。
3. **キャッシュを意識した運転**(プロンプトキャッシュTTL=5分。**適用範囲=指令塔が従量モデルのとき**): 委譲先の完了をブロックで待たず、待ち時間に並行委譲・指示書起草・記録更新を入れる(無ければターンを終えて完了通知で再開)。検収・実読はキャッシュが温かいうちにまとめ打ち。長考停止1回ごとに指令塔の文脈全量(数十万tok)の再書込コストが発生する、が根拠。
4. **盲点の即時反映**: 型・運用の欠陥に気づいたら「直前まで待たず」その場で仕様反映→実装→テストまで行う。

4b. **降板プロトコル(v0.4.1追加)**: 開発が一段落した後の派生対話(承認往復・小修正・文書更新)は委譲ゼロ・対話中心になり、従量指令塔ではターンごとの文脈全量再読(キャッシュTTL 5分切れ)だけが積み上がる — 出力は数k tokでも入力は1ターン数十万tokになり得る。対策 = ①承認済みWPが全て検収・記録済みになったら指令塔は**自分から降板を提案**(またはユーザーの「降板」で発動)②STATUS.md/cost-logへ申し送りを書き切って文脈を外部化 ③ユーザーが**同一セッションで `/model` を枠内モデルに切替**(会話文脈は保持・以後は枠内=従量ゼロ)④以後従量モデルが要る局面はサブエージェントへ蒸留した要約だけ渡す。`/model` での従量復帰は原則しない(全文脈を従量単価で再読するため)。金銭・PII・不可逆の派生作業はP1相当の新WPとして扱い、降板中でも従量モデルの裁定を必須とする。手順の正本は orch skill §8、コスト構造の解説は README「プロンプトキャッシュの仕組み」。
5. **読み込み分担の分岐(ROI)**(「安いモデルへ全振り」はしない): ROI = 判断の質への寄与 × リスク ÷ トークンコスト で分岐する。

**第一パラメータ=既存資産の実測規模**。判定・見積りの前に対象スコープへ
`python3 skills/orch/scripts/scope_size.py <paths>` を1コマンド実行し、推定トークンで帯を決める:

| 実測規模(対象スコープ) | 既定の読み方 | 実測例 |
|---|---|---|
| **~100k tok(≈1万行)** = 直読圏 | 指令塔が必要分を全読してよい。委譲は固定費(~30k+往復)負け | 組込みロボット制御53k・Webアプリsrc 42k — 直読が有効だった(重要バグ3件は直読側で発見) |
| **100k〜500k tok** = ハイブリッド | 広さ=Sonnet並列で圧縮+判断コアを指令塔が直読 | (この帯の実測はまだ少ない) |
| **500k tok超** = 蒸留必須 | Sonnet並列が前提。直読は判断点のピンポイントのみ | 業務自動化基盤 全体1.36M — Sonnetファンアウト運用が有効だった |

注意: 測るのは**触るサブシステム**であってrepo全体ではない(大規模repoでも1区画だけ触るならハイブリッド帯になり得る)。規模帯を土台に、開発フェーズとリスクで下表のとおり上書き補正する:

| 状況 | 指令塔の直読 | Sonnet委譲(蒸留報告) |
|---|---|---|
| **常に(どの状況でも削らない)** | diff目視・受け入れ条件まわり・金銭/PII/不可逆に触る箇所・検収のサンプル実読 | — |
| **新規開発・部分調整**(読むべき既存物が少ない) | 直読が正。トークン影響が小さく、委譲は固定費で負ける | 原則不要 |
| **既存大規模の理解**(リファクタ・全体レビュー・調査) | 判断が架かるコアをピンポイント直読 | 周辺の「広さ」を並列で圧縮 |
| **P1/P2案件**(高リスク) | 厚め。蒸留報告だけで設計・検収判定をしない(lessons/003) | 補助として使う |
| **P3/P4案件**(低リスク・定型) | diff+要点のみ | 広く使ってよい |

原則の一行化: **広さはSonnetで圧縮、深さ(判断点)は指令塔が直読**。コスト削減のために判断の質を落とさない。

## パターン(=モード)

| パターン | モード名 | 適用条件 | 分担 | 指令塔出力の目安/WP |
|---|---|---|---|---|
| **P1** | 安全(厚め) | 高リスク×高複雑(金銭・PII・不可逆)。例: 決済・認証・個人情報を扱う機能 | 指令塔が設計書を書く+指示書+検収 | **8〜12k tok** |
| **P2** | 安全(標準) | 中リスク or 複雑だが定型化可能。例: リファクタ、非コア画面 | 指令塔の指示書(A4半分)→Opus実装→指令塔検収 | **4〜6k tok** |
| **P3** | 割安 | 低リスク×定型。例: サイト移行、CRUD画面 | Opus自走+詰まった時だけ1問1答 | **1〜2k tok** |
| **P4** | 指令塔不要 | 判断ゼロの物量。例: 一括変換、テスト実行、データ移行スクリプト | Sonnet/機械作業、検収もセルフチェック | **≈0** |

- 目安は**差し戻し1回込み**(実運用の初期実績: 検収のあるWPはおおむね1回差し戻しが発生する。これが見積り分散の最大要因)。
- 見積り = WP分解 → 各WPにパターンを割付 → 上の帯を合算。×2精度で事前提示できる。

## パターン判定の型(判定シート方式)

判定インプットは5問の定型シート。**シート埋めはSonnet**(対象ファイルと要件を読んで各問に1行で回答)、判定はスコアで機械分岐し、**迷い域だけ指令塔が1ターンで裁定**(出力数百tok=従量でも数円)。

| # | 問い | Yesなら |
|---|---|---|
| 1 | 金銭が動くか(決済・課金・返金) | +2 |
| 2 | PII・機微情報に触るか | +2 |
| 3 | 不可逆か(削除・公開・DNS・本番DB) | +2 |
| 4 | 依存関係が広いか(3モジュール以上/他システム連携) | +1 |
| 5 | 定型作業か(CRUD・変換・移行・既存パターンの複製) | -2 |

機械分岐: **4点以上=P1 / 2〜3点=P2 / 0〜1点=P3 / -1点以下=P4**。境界±1点や判断に迷う場合のみ指令塔裁定。迷ったら1段安全側(特に金銭・PII絡みは常に安全側)。

## 検収(削らない)

- 3点セット: (1)受け入れ条件の再実行 (2)git status/diffの目視 (3)成果物のサンプル実読。
- 実読では毎回「これを悪用するなら/これが途中で失敗したら」を1分自問する(lessons/003の根拠: 検収で見つかる欠陥は実行部隊のミスではなく**指令塔の設計の穴**が主。実運用初期に認証の本人確認前紐付け経路とWebhook再送の恒久未処理経路の2件を検収実読で捕捉)。
- 検収は入力中心で指令塔出力をほぼ使わない=安い保険。割安モード(P3)でも省略しない。P4のみセルフチェックで代替可。
- 差し戻し時は**先に設計書を改訂してから**修正指示を出す(設計と実装の乖離を作らない)。

## 実績ログ(cost-log)

- WP完了ごとに1行追記: 日付/プロジェクト/WP/パターン/委譲先・実測tok(サブエージェントのツール結果から正確に取れる)/指令塔出力推定tok(成果物サイズから)/差し戻し回数/結果。雛形は orchestration/cost-log.template.md。
- 指令塔自身の消費はセッション内から正確に見えないため**推定+ダッシュボード較正**の二本立て。運用初週は1日1回ダッシュボード実測をメモ→推定係数を較正。
- 蓄積が増えるほどパターン帯の予測精度が上がる。

## 委譲の型(P1〜P3で使う指示書の要件)
- 目的1行 / 受け入れ条件(検証可能な形で) / 前提となる調査済み事実 / 触ってよい範囲・禁止事項 / 完了時の報告形式。A4半分以内。
- 作り込んだscaffoldingは渡さない。検証スクリプトは「テストデータ完全後始末つき」を必須要件にする。

## 承認ゲート(パターンによらず必ず止まる)
- 設計の大きな決定 / 不可逆・破壊的操作 / 外部公開(デプロイ・DNS・公開リポジトリ) / クレジット(従量)投入

## クレジット運用の目安
- 月次予算を決め、超える投入は「何にいくら使うか・代替案」を提示して承認を得る。
- 参考実測: P1×2WP(認証基盤+決済基盤)完了セッションの指令塔従量分は**$2〜4程度**の見込み(出力$50/MTok換算+入力。委譲592k tokは全て枠内)。
