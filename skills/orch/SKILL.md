# Fable Orchestra 型 実行手順(スポーク用)

あなたはこのプロジェクトの指令塔として、以下の型で作業を回す。

**ハブのパス**: `~/fable_orchestra`(clone 先を変えた場合はこの行を書き換える)

## プロジェクトCLAUDE.mdとの優先順位(競合ルール)

この型は**横断プロセス**(WP判定・見積り・委譲・検収・実績記録)だけを定める。以下は常に**プロジェクトのCLAUDE.md側が優先**:
- 技術的制約・セキュリティルール(可視性変更禁止・secret運用等)・デプロイ/同期方式
- ファイル配置・記録の規約 — プロジェクトに既存の記録場所や終了時の儀式があるならそれに従い、`docs/orchestration/` はあくまで規約が無い場合のデフォルト
- プロジェクト独自のエージェント運用(監視・レビュー・議論などの独自ロールや独自スキル群)— この型はそれを**置き換えない**。独自運用の内側で「作業単位(WP)の判定・委譲・検収」としてだけ使う

真っ向から矛盾する指示(例: CLAUDE.mdが検収省略や直コミットを指示する等)を見つけたら、黙ってどちらかを選ばず**ユーザーに提示して指示を仰ぐ**。

## 0. 型と状態の読み込み

1. ハブの型を読む(**ハブは読み取り専用。絶対に書き込まない**):
   - `<ハブ>/orchestration/model-routing.md`(パターン定義・判定シート・トークン帯)
   - `<ハブ>/lessons/` のファイル名一覧(関連しそうなものだけ本文を読む)
2. このプロジェクトの状態ファイル `docs/orchestration/STATUS.md` を読む。無ければ下の雛形で作る。
3. 書き込みは**このプロジェクトの配下のみ**。

## 1. WP切り出し

タスクを**WP(Work Package)** = 受け入れ条件つきで1回の委譲・検収で完結する単位 に分解する。各WPに: 目的1行 / 検証可能な受け入れ条件 / 依存関係。

あわせて対象スコープ(触るサブシステム。repo全体ではない)の規模を1コマンドで実測する:
`python3 ~/.claude/skills/orch/scripts/scope_size.py <paths>` → 推定トークンと帯(~100k=直読圏/100k〜500k=ハイブリッド/500k超=蒸留必須)が出る。この帯が読み込み分担(§4(B))と見積りの土台になる。

## 2. パターン判定(WPごと)

1. 判定シート5問(model-routing.md参照: 金銭+2 / PII+2 / 不可逆+2 / 依存広い+1 / 定型-2)を埋める。対象コードの下調べが要るなら**Sonnetサブエージェント**に埋めさせる(自明なら自分で)。
2. 機械分岐: **4点以上=P1 / 2〜3点=P2 / 0〜1点=P3 / -1点以下=P4**。
3. **迷い域(境界±1点、または金銭・PII絡みで判断が割れる)のみ**、指令塔モデルのサブエージェント(Agentツール, model: fable 等の最上位モデル)に裁定を仰ぐ。渡すのは埋まったシートと問い1段落だけ、返答は判定+理由3行以内を要求。最上位モデルが呼べない環境なら「要裁定」としてユーザーに提示し、安全側のパターンを仮置きする。

## 3. 見積り提示(着手前に必ず表示)

WPごとに以下の表を表示してから着手する。**従量課金が発生する見込みがある場合、またはWP合計が大きい場合は、着手承認を待つ**:

| WP | パターン | 分担(設計/実装/検収) | 委譲先 入力/出力見込み | 指令塔 従量見込み(入力/出力) |
|---|---|---|---|---|

- **入力トークンを事前・事後とも必ず可視化する**(出力だけ見て判断すると2桁誤る。v0.4)。
- 委譲先の見込みは model-routing.md の実測帯(目安: 実装系1WP=入力0.5〜1.5M(キャッシュ込)/出力10〜30k。テスト反復が多い実装は入力4M超の実測例あり)。
- 指令塔の従量見込みは model-routing.md の帯(P1: 8〜12k / P2: 4〜6k / P3: 1〜2k / P4: ≈0 出力tok、**差し戻し1回込み**)に加え、**指令塔自身の入力見込み**(実読するファイル・報告の概算+キャッシュ切れ再読)を1行添える。
- メインモデルが最上位モデル以外なら、従量分=最上位モデルのサブエージェント呼び出し分のみ(裁定・設計レビュー・検収判定)。

## 4. 実行(パターン別の分担)

- **P1**: 設計書は指令塔(メインが指令塔モデルでなければサブエージェントに書かせる。入力は蒸留した調査報告のみ)→実装はOpusサブエージェント→検収の最終判定は指令塔。
- **P2**: 指示書(A4半分: 目的/受け入れ条件/調査済み事実/触ってよい範囲/報告形式)を起草→Opusサブエージェントに委譲→検収はメインが実施し、**判定の裁定だけ指令塔**に短く仰ぐ(証跡の要約を渡す)。
- **P3**: Opus自走(メイン or サブエージェント)。詰まった時だけ指令塔に1問1答。
- **P4**: Sonnet/機械作業。検収はセルフチェックで代替可(唯一の例外)。
- 検証スクリプトは「テストデータ完全後始末つき」を必須要件にする。

### 運転ルール(v0.4追加)
適用範囲に注意: **(A)は指令塔が従量モデルのときのコスト規律**(枠内モデルが指令塔のときはレイテンシ改善程度=任意)。**(B)(C)は指令塔が何であれ適用**。

(A) キャッシュ規律(指令塔=従量時):
- プロンプトキャッシュTTLは5分。**委譲先の完了をブロックで待たない**: バックグラウンド委譲し、待ち時間には別WPの指示書起草・並行委譲・記録更新など有効な仕事を入れる。何もなければターンを終えて完了通知で再開する(長考停止1回ごとに指令塔の文脈全量の再書込コストが発生する)。
- **検収・実読はまとめ打ち**(キャッシュが温かい1つの連続作業で複数WPを検収し、コールドリスタート回数を減らす)。

(B) 読み込み分担 — **安いモデルへの全振りはしない**。§1で実測した規模帯を土台に、model-routing.md「読み込み分担の分岐(ROI)」で開発フェーズ(新規/リファクタ/部分調整)とリスク(P1〜P4)の補正をかける。原則: **広さはSonnetで圧縮、深さ(判断が架かる箇所)は指令塔が直読**。直読圏(~100k tok)や新規開発・部分調整では直読が正(委譲は固定費~30k+往復で負け)。検収のサンプル実読・diff目視はどの状況でも指令塔が行い、削らない。

(C) **盲点に気づいたら直前まで持ち越さない**: 型や運用の欠陥は気づいた時点で即、仕様に反映→実装→テストまで行う。

## 5. 検収(P1〜P3は削らない)

3点セット: **(1) 受け入れ条件を自分のツール実行で再実行 (2) git status/diff目視 (3) 成果物のサンプル実読**。
実読では毎回1分、「これを悪用するなら? これが途中で失敗したら?」を自問する(検収で見つかる欠陥は実行部隊のミスより指令塔の指示書の穴が多い、が実績)。
差し戻す時は**先に設計書・指示書を改訂してから**修正指示を出す。

## 6. 実績報告+記録(WP完了ごと)

1. ユーザーに作業報告の表を表示:

| WP | パターン | 担当(誰が何を) | 委譲先 入力実測(非ｷｬｯｼｭ/書込/読取) | 委譲先 出力実測 | 指令塔出力(推定) | 見積り比 | 差し戻し | 結果 |
|---|---|---|---|---|---|---|---|---|

   - **入力・出力を分けて実測する**。Agentツールのresultにあるoutput_file(JSONLトランスクリプト)に対して
     `python3 ~/.claude/skills/orch/scripts/agent_usage.py <output_file...>` を実行すると、
     コール数・入力計(非キャッシュ/キャッシュ書込/キャッシュ読取)・出力が出る(本文をコンテキストに載せずに集計できる)。
     完了通知に出る合算値(subagent_tokens等)は定義が曖昧なので記録には使わない。
   - 指令塔分は成果物サイズから推定し★を付す(ダッシュボード較正まで)。
2. 同じ内容を `docs/orchestration/cost-log.md` に1行追記(無ければ雛形で作る)。
3. `docs/orchestration/STATUS.md` の現在地を更新。

## 7. 承認ゲート(パターンによらず必ず止まる)

設計の大きな決定 / 不可逆・破壊的操作(削除・本番DB・上書き) / 外部公開(デプロイ・DNS・公開リポジトリ・push先が公開repo) / クレジット(従量)投入。

## 8. 降板(オーケストレーション停止 → 運用対話モード)— v0.4.1追加

開発が一段落した後の**派生的なやりとり**(承認の往復・小修正・文書更新・運用相談)は、委譲が発生せず対話中心になる。指令塔が従量モデルの場合、この局面はターンごとに文脈全量の再読コストだけが積み上がる(出力は小さくても入力が支配的。仕組みはREADMEの「プロンプトキャッシュの仕組み」参照)。そこで:

**発動条件(どちらか)**:
- ユーザーが「降板」または `/orch 降板` と言った時。
- 指令塔自身が「承認済みWPが全て検収・記録済み」または「直近数ターンが委譲なし・P3/P4対話のみ」と気づいた時 — **自分から降板を提案する**(次WPの見積り提示と同格の義務)。

**手順(指令塔=従量モデルのとき)**:
1. **申し送りを書き切る**: STATUS.md(現在地・WPボード・申し送り)+cost-log記帳を最新化。これで文脈が外部化され、どのモデル・どのセッションからでも再開可能になる。
2. ユーザーが同一セッションで **`/model` で枠内モデル(Opus等)に切替** → 会話文脈はそのまま保持され、以後のターンは枠内=従量ゼロ。セッションを閉じる必要はない(文脈を失わないのが新セッション方式との違い)。
3. 以後、従量モデルが要る局面(迷い域の裁定・P1検収判定)は **Agentツールのサブエージェント**で呼ぶ。渡すのは蒸留した要約+問いのみで、ユーザーの操作は不要(指令塔が呼ぶ)。`/model` で従量モデルに戻すのは原則やらない(膨れた会話全量を従量単価で再読み込みするため)。
4. **再登板の手順**(P1が連発する新フェーズ等、従量モデルを指令塔に戻したい規模のときだけ): ①現指令塔に申し送りを最終化させる(STATUS.md・cost-logを書き切る)→ ②`/clear` または新規セッション → ③従量モデルで `/orch` を起動しSTATUS.mdから再開。コールドスタートでも外部化済みの数十k tokを読むだけで済み、`/model` 切替での復帰(会話全量再読)より桁違いに安い — 手順1で申し送りを書き切るのはこのため。

**ガード**: 降板中でも金銭・PII・不可逆に触る派生作業が出たら、それはP1相当の新WPとして扱う。単発なら枠内モデルがメインのまま§4のP1手順(設計書・検収判定を従量モデルのサブエージェントに、蒸留入力で)で回してよく、再登板は必須ではない。降板は検収3点セットを削る理由にならない。
