# Fable Orchestra

**最上位モデル(従量課金)を「指令塔」、サブスク枠内モデルを「実行部隊」に分担させて、AI開発のROIを最適化する運用の型。**

Claude Code の最上位モデル(Fable など)は強力ですが従量課金で割高です。一方、サブスクプラン枠内の Opus / Sonnet は実質「使い放題の固定費」。この構造下で最適化すべき変数はただ一つ — **「高いモデルのトークンを何に使うか」** です。

Fable Orchestra は、この問いに対する実運用から生まれた答えをパッケージ化したものです:

> 指令塔(高いモデル)は「出力が短く判断の重い仕事」= 設計・依存関係の洗い出し・検収**だけ**に使い、コードの物量・調査の読み込みは枠内モデルに流す。

実測例: 認証基盤+決済基盤の2作業を完了したセッションで、実行部隊(Opus/Sonnet)への委譲は約59万トークン(すべて枠内)、指令塔の出力は約1.5万トークン。従量換算で**1セッション$2〜4程度**に収まりました。

## 考え方(3つの柱)

### 1. パターン分担(P1〜P4)

作業を「リスク(金銭・PII・不可逆性)×複雑さ」で評価し、指令塔の関与量を4段階から選びます。

| パターン | モード | 分担 | 指令塔の出力目安/作業 |
|---|---|---|---|
| P1 | 安全(厚め) | 指令塔が設計書+指示書+検収 | 8〜12k tok |
| P2 | 安全(標準) | 指令塔は指示書と検収のみ、実装は枠内モデル | 4〜6k tok |
| P3 | 割安 | 枠内モデル自走、詰まった時だけ1問1答 | 1〜2k tok |
| P4 | 指令塔不要 | 機械作業+セルフチェック | ≈0 |

パターン選択は5問の判定シート(金銭+2 / PII+2 / 不可逆+2 / 依存広い+1 / 定型-2)でスコア化し機械分岐。**迷い域だけ**指令塔が1ターンで裁定します(数円)。詳細は [orchestration/model-routing.md](orchestration/model-routing.md)。

### 2. 検収3点セット(どのモードでも削らない)

**(1) 受け入れ条件を自分のツール実行で再実行 (2) git status/diff 目視 (3) 成果物のサンプル実読**。

実読では「これを悪用するなら? 途中で失敗したら?」を自問します。実運用では、検収で見つかる欠陥の主因は実行部隊のミスではなく**指令塔自身の指示書の穴**でした(セキュリティ経路の見落とし、障害時の再送設計の欠陥など)。検収は入力中心で安く、削ると一番高くつく場所です([lessons/003](lessons/003-inspection-finds-real-bugs.md))。

### 3. 見積り→実績のループ

- **着手前**: 作業(WP)ごとに「パターン/分担/従量見込みトークン」の表を提示(差し戻し1回込みで見積る)
- **完了後**: 「誰が何を担当/実測トークン/見積り比/差し戻し回数/結果」を表示し、cost-log に1行追記
- 実績が溜まるほどパターン帯の予測精度が上がる

## インストール

前提: [Claude Code](https://claude.com/claude-code)

```bash
# 1. ハブとして clone(場所は任意。既定は ~/fable_orchestra)
git clone https://github.com/akiratsukakoshi/fable_orchestra.git ~/fable_orchestra

# 2. スキルをユーザーレベルに配置(全プロジェクトで /orch が使えるようになる)
mkdir -p ~/.claude/skills
cp -r ~/fable_orchestra/skills/orch ~/.claude/skills/

# 3. clone 先を変えた場合のみ、SKILL.md 冒頭のハブパスを書き換える
```

## 使い方

任意のプロジェクトで Claude Code を起動し(メインモデルは Opus 推奨)、

```
/orch <やりたい作業>
```

と打つだけです。あとはスキルが型どおりに進めます:

1. **WP切り出し** — 作業を「受け入れ条件つきで1回の委譲・検収で完結する単位」に分解
2. **パターン判定** — 判定シート→機械分岐、迷い域のみ指令塔裁定
3. **見積り提示** — 従量見込みを表で提示(発生する場合は着手承認を待つ)
4. **実行** — パターン別に委譲。読み込みは規模×フェーズ×リスクのROI分岐で分担(広さ=安いモデルで圧縮、深さ=指令塔が直読)
5. **検収** — 3点セット。差し戻しは「先に設計書を改訂してから」
6. **実績報告** — 分担と実測トークンの表を表示し、プロジェクト内のログに記録

### マルチプロジェクト並行(ハブ&スポーク)

- **ハブ**(この repo の clone)= 型・教訓。**読み取り専用**
- **スポーク**(各プロジェクト)= 状態(`docs/orchestration/STATUS.md`)と実績ログを**自分の配下にだけ**書く

書き込み先が分離されているので、複数プロジェクトのインスタンスを同時に走らせても衝突しません。

### 承認ゲート(パターンによらず必ず止まる)

設計の大きな決定 / 不可逆・破壊的操作 / 外部公開(デプロイ・DNS・公開リポジトリ) / 従量クレジット投入。

## 降板 — 開発が一段落したら指令塔を降ろす(v0.4.1)

開発が一段落した後も、承認の往復・小修正・文書更新・運用相談といった**派生的なやりとり**はしばらく続きます。この局面は委譲が発生せず対話中心なのに、従量モデルが指令塔に座り続けると**ターンごとに会話文脈の全量再読コスト**が積み上がります(仕組みは次節)。そこで型v0.4.1では「降板」を定義しました:

**降板の基本動作**: 承認済みWPが全て検収・記録済みになったら、指令塔は自分から降板を提案します(ユーザーが「降板」/`/orch 降板` と言っても発動)。①STATUS.md・cost-logに申し送りを書き切って文脈を外部化 → ②ユーザーが同一セッションで `/model` を枠内モデルに切替。**会話文脈は保持されたまま、以後のターンは従量ゼロ**になります。

**降板後に従量モデルの判断が要るときの判定**:

| 状況 | やること | セッション操作 |
|---|---|---|
| 1問1答・単発の裁定/検収判定 | 枠内指令塔がサブエージェント(従量モデル)を呼ぶ。渡すのは蒸留した要約+問いのみ | 不要 |
| 単発のP1派生WP(金銭・不可逆) | 枠内メインのまま、設計書・検収判定だけ従量サブエージェントへ(§4のP1手順) | 不要 |
| P1が連発する新フェーズ(再登板) | 申し送りを最終化 → `/clear` or 新セッション → 従量モデルで `/orch` 再開 | 新セッション |

`/model` で従量モデルに**戻す**のは原則やりません — 膨れた会話全量(数十万tok)を従量単価で再読み込みするためです。再登板は外部化済みのSTATUS.md(数十k tok)から読み直す方が桁違いに安く、降板手順①で申し送りを書き切るのはこのコールドスタートを安くするためです。降板中でも検収3点セットは削りません。

## プロンプトキャッシュの仕組み(なぜ派生対話でトークンが溶けるのか)

Claude APIはステートレスで、**毎ターン会話履歴の全量を入力として送り直します**。これを安くするのがプロンプトキャッシュで、要点は4つ:

1. **プレフィックス一致**: キャッシュはプロンプト先頭からの完全一致(バイト単位)。途中が1バイトでも変わると、そこから後ろは全て無効。
2. **TTLは5分**(既定。使うたびに延長。1時間TTLのオプションあり): 5分以内に次のリクエストが来ればキャッシュが効き、跨ぐと切れる。
3. **料金の倍率**: キャッシュ読み取り=基本入力単価の**約0.1倍**/キャッシュ書込=**1.25倍**(5分TTL。1時間TTLは2倍)。
4. **モデル単位**: キャッシュはモデルごとに別。`/model` で切り替えると新モデル側は全量コールドスタート。

**開発フェーズが安い理由**: 指令塔が連続で作業している間はターン間隔が5分以内に収まり、毎ターンの入力のほとんどが「キャッシュ読み(0.1倍)+新規差分の書込」で済みます。

**派生対話が高い理由**: 人間の返信を待つやりとりはターン間隔がほぼ確実に5分を超え、**毎ターンがコールドリスタート=会話文脈全量のキャッシュ書込(1.25倍)**になります。例えば文脈が30万tokに膨れた状態で入力$10/MTokの従量モデルが座っていると、派生の1往復ごとに約 300k × $12.5/MTok ≒ **$3.75**。10往復で$37 — 出力は数百tokの相槌でも、です。同じ問いを蒸留してサブエージェント(入力数k tok)に渡せば数円で済みます。これが「降板」と「検収まとめ打ち」(v0.4運転ルール)の定量的な根拠です。

**公式ドキュメント**(仕組み・料金倍率・TTLの一次情報。仕様は更新されるのでここを正とする):

- プロンプトキャッシュ: https://platform.claude.com/docs/en/build-with-claude/prompt-caching
- 料金(モデル別単価とキャッシュ倍率): https://platform.claude.com/docs/en/pricing
- コンテキストウィンドウ: https://platform.claude.com/docs/en/build-with-claude/context-windows

実測の確認はAPIレスポンスの `usage`(`cache_read_input_tokens` / `cache_creation_input_tokens` / `input_tokens`)で可能。同梱の `agent_usage.py` はこの3値をサブエージェントのトランスクリプトから集計するツールです。

## プロジェクト固有ルールとの共存

スキルは横断プロセス(判定・委譲・検収・記録)だけを定めます。各プロジェクトの CLAUDE.md にある技術的制約・セキュリティルール・独自のエージェント運用は**常にプロジェクト側が優先**。矛盾を見つけたら黙って選ばずユーザーに提示します(スキル内に明文化済み)。

## 更新履歴

### v0.4.1(2026-07-05)— 降板プロトコル

開発が一段落した後の派生対話(承認往復・小修正・文書更新)が従量指令塔に残ると、委譲ゼロなのにターンごとの文脈全量再読だけが積み上がる構造を実運用で特定(人間の返信待ちでキャッシュTTL 5分をほぼ毎回跨ぐため)。対策として「降板」を型に追加: 承認済みWP全完了で指令塔が自分から降板提案 → 申し送りをSTATUS.md/cost-logに書き切り → 同一セッション `/model` 切替で文脈保持のまま従量停止 → 以後の従量判断はサブエージェント(蒸留入力)のみ。再登板の手順と金銭・不可逆ガード込み。詳細は上の「降板」「プロンプトキャッシュの仕組み」節と skill §8。

### v0.4(2026-07-05)— 入力トークンの可視化と読み込みROI分岐

実運用(組込みロボット制御プロジェクトの安定化6WP)で見つかった盲点を型に反映しました。

1. **入力トークンを事前・事後とも可視化** — 委譲先の消費を単一の合算値で記録すると判断を誤ります。実測では、委譲8体の出力計138kに対し**入力計は10.6M**(9割はキャッシュ読み)と2桁の差がありました。見積り表・実績表を「入力(非キャッシュ/キャッシュ書込/読取)/出力」分離に改訂。
2. **実測ツールを同梱** — `skills/orch/scripts/agent_usage.py`: サブエージェントのJSONLトランスクリプトからトークン内訳を、本文をコンテキストに載せずに集計します。
3. **規模実測ツールを同梱** — `skills/orch/scripts/scope_size.py`: 対象スコープの推定トークンから読み方の帯(~100k=指令塔直読圏 / 100k〜500k=ハイブリッド / 500k超=蒸留必須)を1コマンドで判定。**既存資産のボリュームを読み込み分担の第一パラメータ**に位置づけました。
4. **読み込み分担のROI分岐** — 「調査・読み込みは常に安いモデルへ」(v0.3)を撤回。規模×開発フェーズ(新規/リファクタ/部分調整)×リスク(P1〜P4)で分岐し、原則は「**広さは安いモデルで圧縮、深さ(判断が架かる箇所)は指令塔が直読**」。実運用では重要バグの発見は指令塔の直読側に集中しており、全振りは判断の質を落とします。
5. **キャッシュを意識した運転**(指令塔が従量モデルのとき) — プロンプトキャッシュTTL(5分)を跨ぐブロック待ちを避け、委譲はバックグラウンド+完了通知で再開、検収はまとめ打ち。指令塔の従量コストの主因は出力ではなく**キャッシュ切れによる文脈再書込**でした。

v0.3以前: 初版(P1〜P4パターン・判定シート・検収3点セット・ハブ&スポーク)。
