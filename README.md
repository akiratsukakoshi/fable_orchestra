# Fable Orchestra

**最上位モデル(従量課金)を「指令塔」、サブスク枠内モデルを「実行部隊」に分担させて、AI開発のROIを最適化する運用の型。**

Claude Code の最上位モデル(Fable など)は強力ですが従量課金で割高です。一方、サブスクプラン枠内の Opus / Sonnet は実質「使い放題の固定費」。この構造下で最適化すべき変数はただ一つ — **「高いモデルのトークンを何に使うか」** です。

Fable Orchestra は、この問いに対する実運用から生まれた答えをパッケージ化したものです:

> 指令塔(高いモデル)は「出力が短く判断の重い仕事」= 設計・依存関係の洗い出し・検収**だけ**に使い、コードの物量・調査の読み込みは枠内モデルに流す。

実測例: 認証基盤+決済基盤の2作業を完了したセッションで、実行部隊(Opus/Sonnet)への委譲は約59万トークン(すべて枠内)、指令塔の出力は約1.5万トークン。従量換算で**1セッション$2〜4程度**に収まりました。

## 考え方(3つの柱)

### 1. パターン分担(P1〜P4)

作業を「リスク(金銭・PII・不可逆性)×複雑さ」で評価し、指令塔の関与量を4段階から選びます。

| パターン | モード | 分担 | 指令塔の出力目安/作業 |
|---|---|---|---|
| P1 | 安全(厚め) | 指令塔が設計書+指示書+検収 | 8〜12k tok |
| P2 | 安全(標準) | 指令塔は指示書と検収のみ、実装は枠内モデル | 4〜6k tok |
| P3 | 割安 | 枠内モデル自走、詰まった時だけ1問1答 | 1〜2k tok |
| P4 | 指令塔不要 | 機械作業+セルフチェック | ≈0 |

パターン選択は5問の判定シート(金銭+2 / PII+2 / 不可逆+2 / 依存広い+1 / 定型-2)でスコア化し機械分岐。**迷い域だけ**指令塔が1ターンで裁定します(数円)。詳細は [orchestration/model-routing.md](orchestration/model-routing.md)。

### 2. 検収3点セット(どのモードでも削らない)

**(1) 受け入れ条件を自分のツール実行で再実行 (2) git status/diff 目視 (3) 成果物のサンプル実読**。

実読では「これを悪用するなら? 途中で失敗したら?」を自問します。実運用では、検収で見つかる欠陥の主因は実行部隊のミスではなく**指令塔自身の指示書の穴**でした(セキュリティ経路の見落とし、障害時の再送設計の欠陥など)。検収は入力中心で安く、削ると一番高くつく場所です([lessons/003](lessons/003-inspection-finds-real-bugs.md))。

### 3. 見積り→実績のループ

- **着手前**: 作業(WP)ごとに「パターン/分担/従量見込みトークン」の表を提示(差し戻し1回込みで見積る)
- **完了後**: 「誰が何を担当/実測トークン/見積り比/差し戻し回数/結果」を表示し、cost-log に1行追記
- 実績が溜まるほどパターン帯の予測精度が上がる

## インストール

前提: [Claude Code](https://claude.com/claude-code)

```bash
# 1. ハブとして clone(場所は任意。既定は ~/fable_orchestra)
git clone https://github.com/akiratsukakoshi/fable_orchestra.git ~/fable_orchestra

# 2. スキルをユーザーレベルに配置(全プロジェクトで /orch が使えるようになる)
mkdir -p ~/.claude/skills
cp -r ~/fable_orchestra/skills/orch ~/.claude/skills/

# 3. clone 先を変えた場合のみ、SKILL.md 冒頭のハブパスを書き換える
```

## 使い方

任意のプロジェクトで Claude Code を起動し(メインモデルは Opus 推奨)、

```
/orch <やりたい作業>
```

と打つだけです。あとはスキルが型どおりに進めます:

1. **WP切り出し** — 作業を「受け入れ条件つきで1回の委譲・検収で完結する単位」に分解
2. **パターン判定** — 判定シート→機械分岐、迷い域のみ指令塔裁定
3. **見積り提示** — 従量見込みを表で提示(発生する場合は着手承認を待つ)
4. **実行** — パターン別に委譲。調査・大量読み込みは常に安いモデルへ
5. **検収** — 3点セット。差し戻しは「先に設計書を改訂してから」
6. **実績報告** — 分担と実測トークンの表を表示し、プロジェクト内のログに記録

### マルチプロジェクト並行(ハブ&スポーク)

- **ハブ**(この repo の clone)= 型・教訓。**読み取り専用**
- **スポーク**(各プロジェクト)= 状態(`docs/orchestration/STATUS.md`)と実績ログを**自分の配下にだけ**書く

書き込み先が分離されているので、複数プロジェクトのインスタンスを同時に走らせても衝突しません。

### 承認ゲート(パターンによらず必ず止まる)

設計の大きな決定 / 不可逆・破壊的操作 / 外部公開(デプロイ・DNS・公開リポジトリ) / 従量クレジット投入。

## プロジェクト固有ルールとの共存

スキルは横断プロセス(判定・委譲・検収・記録)だけを定めます。各プロジェクトの CLAUDE.md にある技術的制約・セキュリティルール・独自のエージェント運用は**常にプロジェクト側が優先**。矛盾を見つけたら黙って選ばずユーザーに提示します(スキル内に明文化済み)。
