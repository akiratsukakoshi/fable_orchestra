# インプットトークンの仕組み — 「読み込み」にお金がかかる本当の理由

2026-07-07 / Fable Orchestra プロジェクトの実測をもとにした概説(Claude API の場合)。
アウトプット(AIの発言)は指示で短くできるが、**コストの本丸はインプット(読み込み)側**にある。
その「インプット」が何でできていて、どんなロジックで課金されるかを、実測例つきで説明する。

---

## 1. 大前提: AIは毎回「記憶喪失」から始まる

チャットの見た目は「会話が続いている」ように見えるが、裏側のAPIは**ステートレス**(記憶を持たない)。
1回応答するたびに、**会話の最初からの全文をゼロから読み直している**。

```
1往復目: [システム説明+質問1] を読む → 答える
2往復目: [システム説明+質問1+答え1+質問2] を全部読み直す → 答える
3往復目: [同上+答え2+質問3] を全部読み直す → 答える …
```

つまりインプットトークンは「新しく読んだ量」ではなく、**「その時点の会話全文 × 往復回数」の延べ数**で積み上がる。
これがこの文書でいちばん大事な1行。

- 会話全文のことを「**文脈(コンテキスト)**」と呼ぶ
- 文脈は**追記専用**: 一度入ったものは、そのセッション中は毎ターン読み直され続ける(後から抜けない)
- AI自身の発言も、次のターンからは「読み直す対象」= インプットに変わる

### 文脈には何が入っているか

| 中身 | 例 | いつ入るか |
|---|---|---|
| 固定部(オーバーヘッド) | システムプロンプト、ツールの説明書、CLAUDE.md、常設メモリ | セッション開始時に自動で(数万トークン規模) |
| ユーザーとの往復 | 指示、質問、返答 | 会話のたび |
| ツールの実行結果 | 読んだファイルの中身、コマンドの出力、検索結果 | ツールを使うたび |
| AI自身の発言 | 説明、報告書、書いたコード | AIが話すたび |

「コードベースが大きいから高い」は半分誤解で、**大量のファイル読みを文脈に入れなければ**リポジトリの大きさはコストにほぼ比例しない(当プロジェクト実測: リポジトリ規模30倍差でもセッション単価は同レンジ)。

---

## 2. 「インプットトークン」は実は3種類ある

毎ターン全文を定価で読み直したら破産する。そこで**プロンプトキャッシュ**という割引の仕組みがあり、インプットは3つの単価に分かれる。

| 種類 | 何のこと | 単価(基準比) | Fable 5 の実額(/100万tok) |
|---|---|---:|---:|
| 非キャッシュ入力 | キャッシュを使わず定価で読んだ分 | 1.0倍 | $10.00 |
| **キャッシュ書込** | 今回新しく文脈に加わった分を、次回のために保存する費用 | **1.25倍** | $12.50 |
| **キャッシュ読取** | 前回までに保存済みの文脈を読み直す費用 | **0.1倍** | $1.00 |
| (参考)出力 | AIの発言 | — | $50.00 |

(他モデルも倍率は同じ。基準単価はOpus 4.8=$5/$25、Sonnet 5=$3/$15、Haiku 4.5=$1/$5)

### キャッシュのイメージ: 喫茶店のキープボトル

- 初回、ボトル(=会話の全文)を入れるときは**少し割高**(1.25倍)で買う = **書込**
- 次回以降、同じボトルから注ぐのは**10分の1の値段**で済む = **読取**
- ただしキープの期限は**5分**。5分以上間が空くとボトルは廃棄され、**もう一度割高で買い直し** = **キャッシュ切れ再書込**
  (オプションで1時間キープもできるが、その場合の買値は2倍)

### キャッシュの効き方は「前方一致」

キャッシュは会話の**先頭からの完全一致**でしか効かない。先頭に近い部分(システムプロンプト等)を1文字でも変えると、それ以降の保存が全部無効になり、書き直しになる。だから「固定の説明は前に、毎回変わるものは後ろに」が設計の原則になっている。

そして重要: **キャッシュはモデル単位**。会話の途中でモデルを切り替えると、それまでの全文が新モデルにとって「初見」になり、全量が書込(1.25倍)で再課金される。

---

## 3. 課金ロジックを1往復で追う

1往復ごとに起きることはこれだけ:

```
今回のコスト =
    保存済みの文脈を読み直す   … 文脈量 × $1/M   (読取)
  + 新しく増えた分を保存する   … 増分 × $12.5/M  (書込)
  + AIが話す                  … 発言量 × $50/M  (出力)
```

例: 文脈が10万トークンある状態での1往復(Fable 5)
- 読取 100k × $1/M = **$0.10**
- 書込(増分3kとして) 3k × $12.5/M = **$0.04**
- 出力(1.5kとして) 1.5k × $50/M = **$0.08**
- → **1往復 ≈ $0.2〜0.25**。これが30〜60回積み重なるのが1セッション。

つまりセッションのコストはほぼ **「往復数 × その時点の文脈量」** で決まる。
「読取◯百万トークン」という数字に驚く必要はない — それは同じ文脈を何十回も読み直した**延べ数**であり、割引(0.1倍)が効いた結果の姿。

## 4. 実測例(当プロジェクト、2026-07-07)

Fable 5 を指令塔にした開発セッション(3つの作業パッケージを完遂)の実測:

| 項目 | トークン | 金額 |
|---|---:|---:|
| キャッシュ読取(平均文脈 約13万 × 68往復の延べ) | 10,662k | 約$10.7 |
| キャッシュ書込(新規に文脈へ入った分+キャッシュ切れ再書込) | 405k | 約$5.1 |
| 非キャッシュ入力 | 8k | 約$0.1 |
| 出力(AIの発言) | 82k | 約$4.1 |
| **合計** | | **$19.90** |

ここから読み取れること:

1. **出力は全体の2割**。インプット側(読取+書込)が8割を占める — 「アウトプットを短くする」だけでは効かない理由
2. 読取$10.7は、もしキャッシュがなければ**$107**だった。10分の1割引が効いている
3. 同じセッション内でも、**会話が長くなるほど1往復の単価が上がる**。実測では序盤の1往復$0.24 → 終盤$0.42(文脈が11万→18万に育ったため)
4. キャッシュ切れ(5分放置)の再書込は1回$1〜4級。太った文脈ほど買い直しが高い

## 5. だから、コストを下げる打ち手はこうなる

「トークンを削る」ではなく「**往復数と文脈の育ち方を設計する**」のが本筋。

| 打ち手 | 効く理由 |
|---|---|
| **独立した読み・記録を1往復に束ねる** | 往復数が減る = 全文読み直しの回数が減る |
| **5分以上待つ作業は待たずに他の仕事を入れる**(または待った後の再開を1往復にまとめる) | キャッシュ切れの買い直し回数・直後の細切れ往復を減らす |
| **大量のファイル読みは安いモデルに任せ、要約だけ受け取る** | 高いモデルの文脈に生データを入れない = 以降の全往復の読取が軽い(複利で効く) |
| **区切りごとに状態をファイルに書き出し、セッションを畳む** | 文脈をゼロリセットできる。ただし新セッションの立ち上げ+終了に固定費($2〜3)があるので、**続きが短い作業なら続行が正解** |
| **AI自身の発言も簡潔に** | 発言は次のターンから毎回読み直される「インプット」になる |
| **会話途中のモデル切替をしない** | キャッシュはモデル単位。切り替えた瞬間に全文が割高な書込で再課金 |

逆に、効かない・誤解しやすいもの:

- 「ファイルに書き出せば読み直されない」→ ✕。一度会話に出た内容はセッション内では抜けない(効くのはセッションの境界をまたぐとき)
- 「読取トークンの絶対値が大きい=無駄」→ ✕。多くの場合それはキャッシュ割引が効いている証拠。見るべきは**往復数×平均文脈**と**キャッシュ切れ回数**

## 6. なぜサブスク利用者は気づかないのか

Claude のサブスクリプション(Pro/Max)では、この積算が**定額の中に隠れている**。裏側では全く同じロジックでトークンが積み上がっているが、請求が変動しないので意識する機会がない。従量課金(API)に移った瞬間、「1往復ごとに会話全文を読み直している」という構造がそのまま請求書になって現れる — 本文書の内容は、その請求書を読むための地図である。

### ダッシュボード/ usage欄の読み方(おまけ)

- `input_tokens` は**非キャッシュ分だけ**。プロンプト全体の大きさは `input + cache_creation + cache_read` の合計
- `cache_read_input_tokens` がずっとゼロなら、どこかで先頭の文字列が毎回変わっていてキャッシュが壊れている(タイムスタンプの埋め込み等が典型)
