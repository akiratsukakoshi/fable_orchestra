# 005: セッション境界の経済学(続行/切替/避難の判定) — v1.0採用済み

**従量指令塔のコストは「往復数×その時点の文脈」で積算されるため、セッション境界の設計が$数ドル/WP級で効く。ただしプロンプトキャッシュはモデル単位なので、素朴なモデル切替による立ち上げ費の避難は成立しない。成立するのは「蒸留を挟む」か「従量モデルを器なしのサブエージェントで呼ぶ」の2形態のみ。**

状態: **v1.0採用済み**(2026-07-08。標本3=従量サブエージェント課金の実測が揃い、判定表を model-routing.md 改訂点8+SKILL §8b に採用。P1のサブエージェント構成のみ任意実験として残存)

## 事実(1セッションの区間実測、2026-07-07)

- 同一セッション3区間: **WP1(P1)=35req/$8.33(平均文脈114k) → WP2群=16req/$6.66(178k、$/req+75%) → 終了記帳=16req/$4.61(セッションの24%)**。セッション計68req/$19.90、3WP完遂=$6.5/WP、差し戻し0。
- 同一セッション続行の割増 ≈ (現文脈−新セッション基底)×往復数×$1/M+キャッシュ切れ再書込の肥大分(178k文脈では$2.2級/回、基底なら$0.6級)。
- 新セッションの固定費(オーバーヘッド)≈ **$2〜2.5/回**: 立ち上げ(基底60〜80kの書込×1.25+状況把握の往復)≈$1+終了(記帳・申し送り)≈$1〜1.5。
- 帰結: **16req級の小WPは続行が正解(損益分岐)。切って得するのは次WPが30req級以上、または現文脈が基底の2倍(≈150k)超のとき。**

## 学び(仕組み)

1. **キャッシュはモデル単位+履歴は器に残る**: `/model` 切替の初回リクエストで、それまでの履歴全量が新モデルのキャッシュ書込(×1.25)として再課金される。「安いモデルに生のSTATUS/指示書を読ませてから従量モデルへ切替」は避難にならず、読みが太い分むしろ膨らみ得る。
2. 効く避難は2形態だけ:
   - **(a) 蒸留渡し**: 枠内モデルが生60〜80kを5〜10kに圧縮してから従量モデルの文脈へ。立ち上げ書込だけでなく以降の全往復の再読基底が下がるため複利で効く(30往復で≈$3)。
   - **(b) 従量モデルのサブエージェント(器なし)**: 枠内モデルが指令塔の器を持ち(¥0)、裁定・検収判定の局面だけ蒸留パケット(15〜30k)でAgentツール経由で従量モデルを呼ぶ。従量モデルは立ち上げ費も履歴積算も持たない(= SKILL §8手順3の構成)。
3. **譲れない線**: 検収の一次情報(diff・テスト出力・設計書の当該箇所)は要約せず生でパケットに入れる。判断の根拠まで要約越しにすると検収品質を毀損する。避難できるのは「状況把握・網羅読み・所在確認」まで。
4. **入口とWPのモデルは別物にできる**: P1〜P4判定は元々安いモデルの仕事(5問シート)なので、入口は常に枠内モデルで開き、P1と判明した時だけ従量モデルへエスカレーションすればよい(申し送りの外部化でコールドスタート$1級)。逆(従量で入ってP3と判明)の方が高くつく。

## 標本3実測(2026-07-08、業務自動化基盤の財務ダッシュボード機能・実装2WP+配備)

上記(b)構成の課金と品質を実測した:

1. **従量サブエージェント(器なし)の課金実測**: 検収裁定1呼び出し=非キャッシュ5.2k+書込27.2k+**読取0**+出力7.4k=**$0.76**。読取0が「器を持たない」ことの実証 — 履歴の積算も立ち上げ費もなく、課金=渡した蒸留パケットの書込+出力のみ。学び2(b)の理論値どおり。
2. **運転の効き**: 一次情報(diff・テスト出力・受け入れ条件の実行結果)を生でパケットに同梱した結果、**tool_uses=0の一発回答・追加往復ゼロ**。「譲れない線」(学び3)を守ることが品質だけでなくコスト面でも最適だった(往復が生まれない)。
3. **検収品質**: 枠内メインの3点セット+従量裁定の二段で、裁定が条件3件を上乗せ(枠内メインの検収が拾えていなかったレース条件など)。差し戻し0・検収発見は継続。**検収される側が検収材料を選ぶ「信頼構造の逆転」は、一次情報を生で渡す規律がある限り今標本では問題化せず**。
4. **対照実験**: 同セッション前半は従量メインで回り、57calls/$19.95(見積り比+85〜270%超過。原因=実装直読+VPS配備・実疎通まで指令塔が実施)。降板→(b)構成に切り替えた後半の検収・配備・修正 約30callsは枠内¥0。**新知見: 配備・実疎通・修正適用のフェーズは降板後(枠内)に回すのが正。**

## 判定表(v1.0採用。正本は SKILL §8b / model-routing 改訂点8)

| **P2/P3のWP** | **枠内モデルがメイン+従量モデルはサブエージェント裁定(既定)** — 実測: 裁定$0.76/回・検収品質維持 |
| **VPS配備・実疎通・修正適用のフェーズ** | **降板後(枠内メイン+従量裁定)に回す** — 実測: 該当約30callsが¥0化 |
